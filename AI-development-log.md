# AIを開発の相棒にした個人開発とリファクタリング

就職活動用に個人開発している2人対戦アクションゲーム（C++ / DxLib）で、AIコーディングアシスタントを「開発の相棒」として活用した記録です。

## きっかけ

ゲーム自体は完成して遊べる状態になっていましたが、中心となるクラス（`SceneGame`）に処理が集中して肥大化し、機能の追加・修正がしづらくなっていました。

「設計をきちんと学び直したい」「話題のAIを、実際の開発でどこまで使えるか試したい」という2つの思いから、AIアシスタントを開発に取り入れてみることにしました。

## 試したこと / どこまでやったか

単にコードを生成させるのではなく、**「なぜその設計が良いのか」を質問して理解してから、自分で実装・調整する**進め方を徹底しました。

- 設計の相談と、デザインパターン（Strategy パターン）の学習
- 毎ラウンド変化する十数種類の「妨害」システムを、巨大な `if` 分岐から **妨害ごとのクラスへ全面リファクタリング**
- 原因の見えないバグ（分身がジャンプしない不具合など）の調査・修正
- 死にコードの除去、重複処理の共通化、画面サイズのマジックナンバーを定数へ集約、といったコード整理
- 作業はすべて GitHub のコミット履歴と設計資料に残し、各段階でビルドを通して動作を確認

## 効果（楽になった・時短になった点）

- 1人では何日も悩んでいた**設計の判断や、原因の見えないバグ調査が短時間で進み**、学習と開発が大幅に時短になりました。
- リファクタリングの結果、**新しい妨害の追加が「クラスを1つ足すだけ」**になり、拡張のコストが大きく下がりました。
- 「動けばよい」で止めず、**保守しやすい構造まで到達**できました。
